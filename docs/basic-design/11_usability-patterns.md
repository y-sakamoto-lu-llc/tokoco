## 1. レスポンシブデザイン（ブレークポイントと切り替えパターン）

### 1.1 ブレークポイント定義

Tailwind CSS の標準ブレークポイントをモバイルファーストで使用する。

| ブレークポイント | 画面幅 | 代表端末 | 主な適用 |
|---|---|---|---|
| （デフォルト） | 0px〜639px | iPhone SE（375px）・iPhone 14（390px） | モバイルレイアウト |
| `sm` | 640px〜767px | 小型タブレット縦向き | モバイルレイアウト（ほぼ同じ） |
| `md` | 768px〜1023px | タブレット横向き | 2カラム・サイドバー切り替え |
| `lg` | 1024px〜1279px | 小型デスクトップ | 3カラム・フルレイアウト |
| `xl` | 1280px〜 | デスクトップ | コンテンツ幅を最大1280pxに制限 |

**設計ルール:**
- コンテナの最大幅: `max-w-screen-xl mx-auto`（1280px）
- モバイルで動作することを先に確認し、`md:` / `lg:` プレフィックスで上位画面幅を追加する

### 1.2 主要画面のレイアウト切り替え

#### ホーム（マイリスト）/ S-06

```
モバイル（〜767px）:
  [ヘッダー]
  [フィルタボタン] ← タップでShopFilterSheet（ボトムシート）を開く
  [1カラム店舗カードリスト]
  [ボトムナビゲーション fixed]

デスクトップ（768px〜）:
  [ヘッダー]
  [ShopFilterBar] ← 横一列でフィルタを常時表示
  [2カラム店舗カードリスト（md）/ 3カラム（lg）]
  ※ ボトムナビゲーションなし。左サイドバーナビゲーション
```

#### 認証画面（ログイン・新規登録）/ S-02・S-03

```
モバイル・デスクトップ共通:
  中央配置カード（max-w-sm）
  背景: neutral-50
```

#### イベント共有ページ / S-12

```
モバイル（〜767px）:
  シングルカラム・フルスクリーン
  ニックネーム → 候補店舗 → 投票ボタンの縦並び
  投票ボタンは w-full（全幅）

デスクトップ（768px〜）:
  max-w-lg の中央配置カード
```

### 1.3 コンテンツ幅の基準

| 用途 | クラス | 幅 |
|---|---|---|
| ページコンテナ（全画面） | `max-w-screen-xl mx-auto px-4 md:px-8` | 〜1280px |
| フォーム・認証カード | `max-w-sm mx-auto` | 〜384px |
| イベント共有ページ | `max-w-lg mx-auto` | 〜512px |
| ダイアログ | `max-w-md` | 〜448px |

## 2. タッチUI

### 2.1 タップ領域の最小サイズ

インタラクティブ要素のタップ領域は最小 **44×44px** を確保する（WCAG 2.5.5 Target Size の推奨値・UX-01）。

| 要素 | 対応方法 |
|---|---|
| プライマリボタン | `h-11`（44px）以上を標準とする |
| アイコンボタン（共有リンクコピー等） | `min-h-[44px] min-w-[44px]` を適用 |
| タグの削除ボタン（「×」） | 表示サイズは小さくても、padding で44×44pxの透明領域を確保（`p-2 -m-2`） |
| ボトムナビゲーションのタブ | `h-14`（56px）以上。アイコン + ラベルで構成 |
| チェックボックス（投票候補選択） | `<label>` 要素で候補店舗全体をタップ領域にする（06_ui-components.md セクション7.2参照） |

**実装パターン（アイコンボタンの透明padding拡張）:**

```tsx
<button
  className="relative p-2 -m-2 rounded-sm focus-visible:ring-2 focus-visible:ring-primary-500"
  aria-label="タグ名 を削除"
>
  <X className="h-3 w-3" />
</button>
```

### 2.2 スワイプ操作の方針

v1.0ではカスタムスワイプジェスチャーは実装しない。

- ブラウザのネイティブスクロール挙動（縦スクロール）に委ねる
- ボトムシート（ShopFilterSheet）は shadcn/ui の `Sheet` コンポーネントを使用する。Sheet は外側タップでクローズする
- 横スワイプによるカード削除等のジェスチャーはv2.0以降で検討する

### 2.3 モバイルキーボード対応

フォーム入力時にソフトウェアキーボードが表示される際の対応:

- 投票フォームのニックネーム入力欄は `inputMode="text"` を指定する（デフォルト動作）
- 数値のみの入力欄（将来的な電話番号等）は `inputMode="tel"` を指定する
- フォームの送信後、`blur()` を呼び出してキーボードを閉じる

## 4. 3タップ以内の操作フロー設計（UX-02）

### 4.1 店舗追加フロー（UX-02 対応）

要件 UX-02: 店舗追加の操作ステップを3タップ以内（店名/URL入力 → 候補選択 → 保存）で完了できる。

```
タップ1: ホーム画面の「店舗を追加」ボタン（FAB または CTA ボタン）をタップ
         → /shops/new に遷移
         → 店名入力欄に自動フォーカス（autoFocus属性）

入力: 店名または URL を入力（キーボード操作のため「タップ」カウントに含まない）

タップ2: 候補リスト（Places API）から店舗名をタップ
         → 住所・カテゴリ・評価等が自動補完

タップ3: 「保存する」ボタン（w-full 全幅）をタップ
         → 保存完了 → ホーム画面に遷移 + 成功トーストを表示
```

**実装上の注意点:**

- 店名入力後の候補リストは `debounce（300ms）` で API を呼び出し、タイプ中のリクエスト数を抑える
- 自動補完後の「保存する」ボタンは `autoFocus` ではなくユーザーによるスクロール操作後にタップする。ボタンは画面下部の見えやすい位置（sticky等）に配置しない。フォームの最下部にw-full配置し、自動補完後はボタンが見えるよう自動スクロールする
- 手動入力（SHOP-05）時は候補選択のタップが省略され2タップで完了

### 4.2 投票フロー（UX-03 対応）

要件 UX-03: 共有リンクへのアクセスから投票完了まで会員登録なしで完結できる。

```
ステップ1: 共有リンク（/events/share/[token]）にアクセス（タップ不要・URLを開く操作）

ステップ2: ニックネームを入力

タップ1: 投票したい候補店舗（チェックボックス付き）をタップ
         ※ 複数タップ可（それぞれ1タップとカウント）

タップ2: 「投票する」ボタン（w-full 全幅）をタップ
         → 完了 → 投票結果画面を表示
```

**ゲスト体験の配慮:**

- 会員登録ページへの誘導は投票完了後のみ表示する（投票前に表示しない）
- エラー時のメッセージは「もう一度試してください」等のシンプルな文言にする

## 5. 空状態（Empty State）の表示パターン

### 5.1 EmptyState コンポーネント

`src/components/ui/EmptyState.tsx` を使用する（06_ui-components.md セクション2参照）。

**props 定義:**

```typescript
type EmptyStateProps = {
  icon: React.ReactNode;         // アイコン（例: <Store className="h-12 w-12" />）
  title: string;                 // 見出し（例: 「まだ店舗が登録されていません」）
  description?: string;          // サブテキスト（任意）
  action?: {
    label: string;               // CTAボタンのラベル
    href?: string;               // リンク遷移先（hrefまたはonClickを指定）
    onClick?: () => void;
  };
};
```

**レイアウト:**

```
中央揃え（text-center）
アイコン（neutral-300色, h-12 w-12）
↓ space-y-3
見出し（text-lg font-semibold text-neutral-700）
↓
サブテキスト（text-sm text-neutral-500）
↓ mt-6
CTAボタン（primary, md:w-auto w-full）
```

### 5.2 画面別の空状態パターン

| 画面 | 条件 | アイコン | タイトル | サブテキスト | CTA |
|---|---|---|---|---|---|
| 店舗一覧（S-06）・フィルタなし | 店舗登録0件 | `Store` | まだ店舗が登録されていません | 気になるお店を追加してみましょう | 「店舗を追加する」→ `/shops/new` |
| 店舗一覧（S-06）・フィルタあり | フィルタ条件に一致する店舗なし | `SearchX` | 条件に一致する店舗が見つかりません | フィルタ条件を変えてみてください | 「フィルタをリセット」 |
| イベント一覧（S-09） | イベント作成0件 | `CalendarDays` | まだイベントがありません | 友人と一緒に行くお店を選んでみましょう | 「イベントを作成する」→ `/events/new` |
| イベント詳細・投票結果（S-11） | 投票0件 | `Vote` | まだ投票がありません | 共有リンクを友人に送りましょう | なし |

### 5.3 フィルタ条件適用時の注意事項

- フィルタ適用中は空状態のアイコン・テキストを「フィルタに一致する結果なし」用に切り替える
- 「フィルタをリセット」CTAのURLは `?` 以降のクエリパラメータを削除したルートパス（`/shops`）にする

## 6. ローディング状態の使い分け

### 6.1 スケルトンローダー vs スピナーの使い分け

| 状況 | 使用する表示 | 理由 |
|---|---|---|
| ページ初期ロード（Server Component のデータフェッチ） | スケルトンローダー | レイアウトシフトを防ぎ、体感速度を向上させる |
| フォーム送信中 | ボタンのDisabled + テキスト変更（「保存中...」） | スペース効率が高く、操作の文脈が明確 |
| 追加データの取得（手動更新ボタン押下） | スピナー（ボタン内アイコン） | 既存コンテンツを残したまま更新を示す |
| ページ遷移中 | Next.js のデフォルト（loading.tsx） | App Router の標準機能で対応 |
| Realtime 購読の初期化 | スケルトンローダー | 投票結果の初期表示まで骨格を表示 |

### 6.2 スケルトンローダーの使用場面

Next.js App Router の `loading.tsx` と React `Suspense` を組み合わせる。

| ファイル | 対象 |
|---|---|
| `src/app/(app)/shops/loading.tsx` | 店舗一覧（ShopListSkeleton: ShopCardSkeleton × 4件） |
| `src/app/(app)/events/loading.tsx` | イベント一覧（EventCardSkeleton × 3件） |
| `src/app/(app)/events/[id]/loading.tsx` | イベント詳細（VoteResultSkeleton） |
| `src/app/events/share/[token]/loading.tsx` | 投票フォーム（投票フォームのスケルトン） |

**スケルトンの原則:**
- 実際のコンポーネントと同一の高さ・幅のスケルトンを用意する（レイアウトシフト防止）
- `aria-busy="true"` をスケルトンコンテナに付与する（A11Y-01対応）
- スケルトンのアニメーションは `animate-pulse` を使用する

### 6.3 フォーム送信中の表示パターン

```tsx
<Button type="submit" disabled={form.formState.isSubmitting}>
  {form.formState.isSubmitting ? (
    <>
      <Loader2 className="mr-2 h-4 w-4 animate-spin" />
      保存中...
    </>
  ) : (
    "保存する"
  )}
</Button>
```

**ルール:**
- 送信中は `disabled` にして二重送信を防ぐ
- テキストを「〇〇中...」に変更してフィードバックを与える
- `Loader2` アイコン（Lucide）の `animate-spin` でスピナーを表示する
- 完了後は成功トーストを表示して画面遷移またはフォームリセットを行う

### 6.4 エラー後の再試行UX

- ネットワークエラー・500エラーはトースト通知で表示し、フォームはそのまま保持する（入力内容を消さない）
- トーストのエラーメッセージ: 「通信エラーが発生しました。もう一度お試しください」
- 送信ボタンは `disabled` を解除して再送信可能にする

## 7. フォームのUXパターン

### 7.1 レイアウト方針

- フォームは常にシングルカラムレイアウトとする（デスクトップでも横並び入力欄は作らない）
- 入力幅はフォームコンテナ幅（最大 `max-w-sm`）に合わせて `w-full` を適用する
- フォーム全体のスペーシング: `space-y-5`（各フィールド間 20px）

### 7.2 インラインバリデーション

バリデーションエラーの表示タイミング:

| タイミング | 動作 |
|---|---|
| フォーム送信時 | 全フィールドを一括バリデーション。エラーがある場合は最初のエラーフィールドにスクロール |
| フィールドのblur時（フォーカスを外した時） | そのフィールドのみバリデーション実行（`mode: "onBlur"`） |
| 入力中（onChange） | 送信後のみ変更を反映（`reValidateMode: "onChange"`） |

**React Hook Form の設定:**

```typescript
const form = useForm<T>({
  resolver: zodResolver(schema),
  mode: "onBlur",
  reValidateMode: "onChange",
});
```

**エラーフィールドへの自動スクロール:**

```typescript
async function onSubmit(data: T) {
  // ...
}

// エラー時は最初のエラーフィールドにスクロール
function onError(errors: FieldErrors<T>) {
  const firstErrorKey = Object.keys(errors)[0];
  document.querySelector(`[name="${firstErrorKey}"]`)?.scrollIntoView({
    behavior: "smooth",
    block: "center",
  });
}

// <form onSubmit={form.handleSubmit(onSubmit, onError)}>
```

### 7.3 フォームの成功・失敗フィードバック

| 操作結果 | フィードバック方法 |
|---|---|
| 保存成功（画面遷移あり） | 遷移後にトースト（success）を表示。次のページで `toast.success("店舗を登録しました")` |
| 保存成功（画面遷移なし・編集等） | フォーム内のインラインメッセージ + トースト（success） |
| バリデーションエラー | `FormMessage` でフィールド直下に赤テキスト表示（送信ボタンはenabled維持） |
| サーバーエラー（400・409） | フォーム上部の `Alert`（error variant）に詳細メッセージを表示 |
| ネットワークエラー・500 | トースト（error）。フォームの入力内容は保持 |

**サーバーエラーの表示:**

```tsx
{serverError && (
  <Alert variant="destructive">
    <AlertCircle className="h-4 w-4" />
    <AlertDescription>{serverError}</AlertDescription>
  </Alert>
)}
```

### 7.4 文字数カウンター

最大文字数が設定されているフィールドには文字数カウンターを表示する。

| フィールド | 最大文字数 | 表示 |
|---|---|---|
| ニックネーム（投票時） | 50文字 | 「{n}/50」を入力フィールド右下に表示 |
| イベントタイトル | 100文字 | 「{n}/100」を入力フィールド右下に表示 |
| メモ（店舗） | 1000文字 | 「{n}/1000」をテキストエリア右下に表示 |

文字数が上限に近づいた場合（残り10文字以下）はカウンターの色を `text-warning-500`（amber）に変える。

### 7.5 削除確認ダイアログ

削除操作は必ず確認ダイアログ（`ConfirmDialog`）を表示する（SHOP-21, EVENT-13, AUTH-14）。

**ダイアログの構成:**

```
タイトル: 「{対象名} を削除しますか？」
本文: 「この操作は取り消せません。」（データによっては影響範囲を追記）
ボタン: [キャンセル]  [削除する]（destructive variant）
```

**フォーカス管理:**
- ダイアログが開いたとき、フォーカスは「キャンセル」ボタンに移動する（誤削除防止）
- `Dialog` コンポーネント（shadcn/ui）はデフォルトで `autoFocus` が最初のフォーカス可能要素に適用されるため、「キャンセル」ボタンを「削除する」より先にDOMに配置する

## 8. ディープリンク設計（share_token経由）

### 8.1 URL構造

| URL | アクセス可能ユーザー | 説明 |
|---|---|---|
| `/events/share/[token]` | 全員（ゲスト含む） | イベント共有ページ。投票フォームを表示 |
| `/events/share/[token]/result` | 全員（投票完了後） | 投票結果の閲覧ページ |

### 8.2 share_token アクセス時の挙動

```
1. ユーザーが /events/share/[token] にアクセス

2. サーバー側でshare_tokenを検証:
   - 有効なtokenかつイベントがオープン中 → 投票フォームを表示
   - 有効なtokenかつイベントがクローズ済み → クローズ通知画面を表示
   - 無効なtoken（存在しない） → 404ページを表示

3. 会員ユーザーがアクセスした場合:
   - 認証ガード（middleware）はshare_token URLを認証不要として除外する
   - ニックネーム入力欄に表示名を自動補完（編集可能）
   - 投票完了後の結果画面でオーナーページへのリンクも表示する

4. ゲストユーザーがアクセスした場合:
   - ニックネーム入力欄は空欄
   - 投票完了後の結果画面に「Tokokoに登録する」への任意誘導リンクを表示
```

### 8.3 クローズ済みイベントの表示

```tsx
// クローズ済みイベントにアクセスした場合の表示
<div className="text-center space-y-4">
  <p className="text-2xl font-bold">このイベントは終了しました</p>
  <p className="text-neutral-500">投票受付が終了しています。</p>
</div>
```

### 8.4 OGP・タイトル設定

共有リンクがSNSでシェアされた際のプレビュー表示:

```typescript
// src/app/events/share/[token]/page.tsx

  const event = await getEventByShareToken(params.token);
```

## 9. ページネーション vs 無限スクロール

### 9.1 v1.0 の方針

**ページネーションを採用する（無限スクロールは採用しない）。**

採用理由:
- MVP フェーズでは実装の複雑性を低く保つ
- URLにページ番号が含まれるため、ページリロード後も位置が維持される
- フィルタ・ソートとの組み合わせが単純

### 9.2 ページネーションの仕様

| 項目 | 値 |
|---|---|
| 1ページあたりの表示件数 | 20件 |
| ページネーションUIコンポーネント | shadcn/ui `Pagination`（v1.0では未追加のため必要時に `pnpm dlx shadcn@latest add pagination` で追加） |
| URLパラメータ | `?page=2`（フィルタパラメータと組み合わせ: `?category=和食&page=2`） |
| 最終ページでの処理 | 「次へ」ボタンを `disabled` にする |

### 9.3 初期表示件数の判断

v1.0 MVP ではユーザーの登録件数は少ないと想定する。20件を超える場合にのみページネーションを表示する（20件以下の場合はページネーションUIを表示しない）。

## 10. パンくず・戻るボタンの挙動

### 10.1 ナビゲーション階層

```
ホーム（マイリスト）/shops
  └── 店舗追加 /shops/new
  └── 店舗詳細 /shops/[id]
      └── 店舗編集 /shops/[id]/edit

イベント一覧 /events
  └── イベント作成 /events/new
  └── イベント詳細 /events/[id]
```

### 10.2 戻るボタンの実装方針

**パンくずリストは v1.0 では使用しない。**

代わりに `PageHeader` コンポーネント（`src/components/ui/PageHeader.tsx`）に戻るボタンを設置する。

```typescript
type PageHeaderProps = {
  title: string;
  backHref?: string;    // 指定した場合、左矢印アイコン + 戻るリンクを表示
  // backHref なしの場合はページタイトルのみ表示（トップレベルページ）
};
```

**戻る先の規則:**

| 現在のページ | 戻る先 |
|---|---|
| `/shops/new` | `/shops` |
| `/shops/[id]` | `/shops` |
| `/shops/[id]/edit` | `/shops/[id]` |
| `/events/new` | `/events` |
| `/events/[id]` | `/events` |

**`router.back()` は使用しない。**

理由: 外部サイトからの直アクセスや、共有リンク経由でのアクセス時に意図しないページに戻る可能性があるため、明示的な `href` を指定する。

### 10.3 共有ページ（`/events/share/[token]`）の戻るボタン

共有ページはナビゲーション階層に含まれないため、戻るボタンを表示しない。投票完了後は結果画面のみ表示する（ブラウザのバックボタンは動作するがUIとしては提供しない）。

## 11. タブ・フォーカス管理（A11Y-01）

### 11.1 フォーカス順序の原則

- DOMの論理的な順序でフォーカスが移動するよう設計する
- `tabIndex` の正の値（`tabIndex={1}`, `tabIndex={2}` 等）は使用しない。DOMの順序で自然にフォーカスが移動するよう設計する
- `tabIndex={0}`: インタラクティブでない要素（`div` 等）をタブフォーカスに含める場合のみ使用
- `tabIndex={-1}`: プログラムからフォーカスを当てるが、タブキーのフォーカス順からは除外したい場合に使用

### 11.2 フォーカスの可視化

全インタラクティブ要素にキーボードフォーカス時の視覚的インジケータを設ける。

```css
/* globals.css に追加（ブラウザデフォルトのoutlineを維持しつつブランドカラーで上書き） */
:focus-visible {
  outline: 2px solid #639922;
  outline-offset: 2px;
}
```

shadcn/ui コンポーネントは `focus-visible:ring-2 focus-visible:ring-primary-500 focus-visible:ring-offset-2` が適用されているため追加実装不要。カスタムコンポーネントには明示的に設定する。

### 11.3 モーダル・ダイアログのフォーカス管理

`Dialog`（shadcn/ui = Radix UI）は以下を自動処理するため追加実装不要:

- ダイアログ開いた時: フォーカスをダイアログ内の最初のフォーカス可能要素に移動
- ダイアログ表示中: フォーカスをダイアログ内に閉じ込める（フォーカストラップ）
- ダイアログを閉じた時: フォーカスをトリガー要素（開くボタン）に戻す

### 11.4 動的コンテンツのフォーカス管理

| 状況 | 対応方法 |
|---|---|
| タグ追加後 | タグ入力フィールドにフォーカスを維持する（タグを追加し続けられるように） |
| タグ削除後 | 削除したタグの次のタグの削除ボタン、または入力フィールドにフォーカスを移動する |
| フォームのステップ遷移（店舗追加: 検索 → 確認） | 次のステップの見出しまたは最初の操作要素にフォーカスを移動する |
| エラー発生後（フォーム送信） | 最初のエラーフィールドにフォーカスを移動する（セクション7.2参照） |

### 11.5 スクリーンリーダー向けの追加対応

| 状況 | 対応 |
|---|---|
| ページ遷移後 | `<h1>` にフォーカスを移動する（Next.js App Router は自動対応。`<h1>` の存在を確認する） |
| 非同期コンテンツ更新（投票結果等） | 更新領域に `aria-live="polite"` を付与（06_ui-components.md セクション11.3参照） |
| ローディング中（スケルトン） | コンテナに `aria-busy="true"` を付与。ローディング完了後は `aria-busy="false"` に変更 |
| 現在のナビゲーション項目 | アクティブなナビゲーションリンクに `aria-current="page"` を付与 |

**ナビゲーションの `aria-current` 実装:**

```tsx
<Link
  href="/shops"
  aria-current={pathname === "/shops" ? "page" : undefined}
  className={cn(
    "flex items-center gap-2 px-3 py-2 rounded-md text-sm font-medium",
    pathname === "/shops"
      ? "text-primary-500 bg-primary-50"
      : "text-neutral-500 hover:text-neutral-900"
  )}
>
  <Home className="h-5 w-5" />
  ホーム
</Link>
```

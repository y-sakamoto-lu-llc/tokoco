## 1. システム概要

### 1.1 サービス概要

Tokoco は「気になるスポットの記録」から「友人グループでの場所選び・投票・日程調整」までをひとつの Web アプリで完結させるサービスである。

### 1.2 コアコンセプト

| コンセプト | 説明 |
|---|---|
| ゼロ摩擦の記録 | 店名または URL を入力するだけで店舗情報を自動補完する |
| タグによる整理 | 「行きたい」「スタメン入り」「行った」などのタグを複数付与して店舗を自由に整理できる |
| ワンリンク完結 | 共有リンク1つで投票・日程調整まで完結し、相手の会員登録を不要にする |

### 1.3 スコープ（v1.0）

**対象:** Web アプリケーション（PC・スマートフォンブラウザ対応）

**対象外（将来フェーズ）:**
- iOS / Android ネイティブアプリ
- フリーミアム課金機能
- 多言語対応

## 2. ユーザー種別と権限

| 種別 | 定義 | できること |
|---|---|---|
| **会員ユーザー** | メール＋パスワードで登録済みのユーザー | 全機能を利用可能 |
| **ゲストユーザー** | 共有リンク経由でアクセスした未ログインユーザー | 投票・日程回答のみ可能（閲覧・記録・リスト管理は不可） |

## 3. 機能要件

### 3.1 認証・アカウント管理

#### 3.1.1 会員登録

| # | 要件 |
|---|---|
| AUTH-01 | メールアドレスとパスワードで新規登録できる |
| AUTH-02 | 登録時にメールアドレス確認メールを送信する |
| AUTH-02a | メールアドレス確認が完了するまでアプリの機能を利用できない |
| AUTH-03 | パスワードは8文字以上、英字・数字混在を必須とする |
| AUTH-04 | 表示名（ニックネーム）を登録時に設定できる |

#### 3.1.2 ログイン・ログアウト

| # | 要件 |
|---|---|
| AUTH-05 | メールアドレス＋パスワードでログインできる |
| AUTH-06 | ログイン状態をセッション維持できる |
| AUTH-07 | 複数端末から同一アカウントで同時アクセスできる |
| AUTH-08 | ログアウトできる |

#### 3.1.3 パスワードリセット

| # | 要件 |
|---|---|
| AUTH-09 | パスワードを忘れた場合、メールアドレス宛にリセットリンクを送付できる |
| AUTH-10 | リセットリンクの有効期限は発行から24時間とする |

#### 3.1.4 アカウント設定

| # | 要件 |
|---|---|
| AUTH-11 | 表示名を変更できる |
| AUTH-12 | メールアドレスを変更できる（変更後に再確認メールを送信） |
| AUTH-13 | パスワードを変更できる（現在のパスワードの入力を必須とする） |
| AUTH-14 | アカウントを削除できる（削除前に確認ダイアログを表示する） |
| AUTH-15 | アカウント削除時のデータ処理は以下の方針とする: ①自分が登録した店舗・タグ・イベント（および紐づく投票）はカスケード削除する ②他ユーザーのイベントへの投票は削除せず、`user_id` を NULL・`voter_name` を「退会済みユーザー」に更新して匿名化する（投票数は保持） |

### 3.2 店舗の記録・管理

#### 3.2.1 店舗の追加

| # | 要件 |
|---|---|
| SHOP-01 | **店名**または**URL**を入力して店舗を登録できる |
| SHOP-02 | 店名入力時: Google Places API で候補を検索・選択し、店舗情報を自動補完する |
| SHOP-03 | URL入力時: 対応サービス（Google マップ等）の URL から店舗情報を自動取得する |
| SHOP-04 | 自動補完される情報: 店名・住所・エリア・電話番号・カテゴリ（ジャンル）・価格帯・評価・営業時間・公式サイトURL・Google マップURL・写真 |
| SHOP-05 | 自動補完に失敗した場合または候補が見つからない場合、店名のみで手動登録できる |
| SHOP-06 | 店舗登録時にメモ（自由テキスト）を任意入力できる |

#### 3.2.2 店舗のタグ

店舗にタグを付与することで一覧の整理・絞り込みができる。タグはユーザーが自由に作成・命名できる。

| # | 要件 |
|---|---|
| SHOP-07 | 任意の文字列でタグを新規作成し、店舗に付与できる |
| SHOP-08 | 過去に自分が作成したタグを入力候補として表示し、再利用できる |
| SHOP-09 | 1つの店舗に複数タグを付与できる |
| SHOP-10 | 付与済みのタグを外すことができる |
| SHOP-11 | 1つの店舗にタグは付与しなくてもよい（タグなしでの登録も有効） |
| SHOP-12 | タグ入力欄には「行きたい」「スタメン入り」「行った」をサジェストとして初期表示する（強制ではない） |
| SHOP-13 | 使用店舗数がゼロになったタグは自動的に候補から削除する |
| SHOP-13a | v1.0 ではタグの専用管理画面（リネーム・一括削除）は提供しない。タグのリネームは「既存タグを外して新しいタグを付け直す」操作で代替する |

#### 3.2.3 店舗の閲覧・絞り込み

| # | 要件 |
|---|---|
| SHOP-14 | 自分が登録した店舗の一覧を表示できる |
| SHOP-15 | 一覧はカード形式で表示し、店名・カテゴリ・価格帯・付与タグを表示する |
| SHOP-16 | 一覧を以下の条件で絞り込みできる: ジャンル / 価格帯 / タグ / エリア |
| SHOP-17 | 一覧を登録日・店名（五十音順）でソートできる |
| SHOP-18 | 店舗の詳細ページを表示できる（全補完情報＋メモ＋付与タグを表示） |

#### 3.2.4 店舗の編集・削除

| # | 要件 |
|---|---|
| SHOP-19 | 登録済み店舗のメモ・タグを編集できる |
| SHOP-20 | 登録済み店舗の基本情報（店名・住所等）を手動で上書き編集できる |
| SHOP-21 | 登録済み店舗を削除できる（確認ダイアログを表示する） |

### 3.3 イベント・共有

「イベント」とは、「金曜の飲み会」などの特定の集まりを単位として作成する、投票・日程調整のまとまりである。

#### 3.3.1 イベント作成

| # | 要件 |
|---|---|
| EVENT-01 | 会員ユーザーがイベントを作成できる |
| EVENT-02 | イベントにタイトル（例:「3月の金曜飲み」）と説明を設定できる |
| EVENT-03 | イベント作成時に自分のマイリストから候補店舗を複数選択して追加できる |
| EVENT-04 | 候補店舗は1件以上設定することを必須とする |
| EVENT-05 | イベント作成後も候補店舗を追加・削除できる |

#### 3.3.2 共有リンク

| # | 要件 |
|---|---|
| EVENT-06 | イベント固有の共有リンクを1件発行できる |
| EVENT-07 | 共有リンクはゲストユーザーでもアクセス可能とする |
| EVENT-08 | 共有リンクにアクセスしたユーザーに対し、候補店舗の一覧と投票フォームを表示する |
| EVENT-09 | 共有リンクに有効期限は設けない（v1.0） |
| EVENT-10 | イベントをクローズできる。クローズ後は共有リンクへのアクセスおよび投票ができなくなる |
| EVENT-11a | 共有リンクの再発行は v1.0 では提供しない。内容を変更したい場合はクローズして新規イベントを作成する |

#### 3.3.3 イベント管理

| # | 要件 |
|---|---|
| EVENT-11 | 自分が作成したイベントの一覧を確認できる（クローズ済みイベントも含む） |
| EVENT-11b | イベント一覧をオープン中／クローズ済みでフィルタリングできる |
| EVENT-12 | イベントのタイトル・説明・候補店舗を編集できる |
| EVENT-13 | イベントを削除できる（確認ダイアログを表示する） |

---

### 3.4 投票

| # | 要件 |
|---|---|
| VOTE-01 | 共有リンクにアクセスしたユーザー（会員・ゲスト問わず）が投票できる |
| VOTE-02 | 投票方式: 候補店舗の一覧から気になる店舗を1つ以上選択して送信する（複数選択可） |
| VOTE-03 | 投票時にニックネームを入力する（ゲストは必須、会員は表示名を自動適用） |
| VOTE-04 | イベント作成者は投票状況をリアルタイムで確認できる（Supabase Realtime を使用） |
| VOTE-05 | 投票結果は候補ごとの得票数を表示し、最多得票の店舗をハイライトする |
| VOTE-05a | 参加者（共有リンクアクセス者）は投票送信後に結果画面を表示できる。ただし v1.0 ではリアルタイム更新は行わず、手動更新ボタン（更新アイコン）で最新の集計を取得できる |
| VOTE-05b | v2.0 以降で参加者画面もリアルタイム更新に対応する |
| VOTE-06 | **重複投票の防止はv1.0では実装しない**（性善説で運用。調整さんに準ずる） |

## 4. 非機能要件

### 4.1 パフォーマンス

| # | 要件 |
|---|---|
| PERF-01 | ページの初期表示（LCP）は3秒以内（通常の4G回線を想定） |
| PERF-02 | 店舗検索の候補表示は入力から1秒以内に応答する |
| PERF-03 | 投票・日程回答の送信後、確認表示まで2秒以内に完了する |

### 4.2 可用性

| # | 要件 |
|---|---|
| AVAIL-01 | 目標稼働率: 99.5%/月（Cloudflare インフラの SLA に依存） |
| AVAIL-02 | メンテナンス時はメンテナンスページを表示する |

### 4.3 セキュリティ

| # | 要件 |
|---|---|
| SEC-01 | 全通信を HTTPS（TLS 1.2以上）で行う |
| SEC-02 | パスワードは Supabase Auth の仕組みに従いハッシュ化して保存する（平文保存禁止） |
| SEC-03 | CSRF 対策を実施する |
| SEC-04 | 全入力値に対してサーバーサイドバリデーションを行う |
| SEC-05 | XSS 対策として出力時のエスケープ処理を徹底する |
| SEC-06 | ユーザーは自分のデータのみ参照・編集できる（Row Level Security を Supabase で設定する） |
| SEC-07 | 共有リンクは推測困難なランダムトークン（128bit以上のエントロピー）で生成する |
| SEC-08 | ログイン試行回数を制限する（Supabase Auth のレート制限を活用） |

### 4.4 ユーザビリティ

| # | 要件 |
|---|---|
| UX-01 | スマートフォンブラウザ（iOS Safari / Android Chrome）で快適に操作できるレスポンシブデザインとする |
| UX-02 | 店舗追加の操作ステップを3タップ以内（店名/URL入力 → 候補選択 → 保存）で完了できる |
| UX-03 | 共有リンクへのアクセスから投票完了までをアプリの会員登録なしで完結できる |

### 4.5 アクセシビリティ

| # | 要件 |
|---|---|
| A11Y-01 | WCAG 2.1 Level AA 準拠を目標とする |
| A11Y-02 | フォーム要素に適切なラベルを付与する |
| A11Y-03 | ブランドカラー（`#639922`）使用時に背景色とのコントラスト比が 4.5:1 以上を確保する |

### 4.6 拡張性

| # | 要件 |
|---|---|
| EXT-01 | フロントエンドとバックエンドを API で分離し、将来のネイティブアプリ対応を容易にする |
| EXT-02 | 多言語対応を見据えて UI テキストを i18n 管理する |
| EXT-03 | フリーミアム実装に備え、ユーザーごとのリソース使用量（登録件数・共有数等）を記録できるデータ構造にする |

## 5. データモデル概要

### エンティティ一覧

```
※ 認証情報（メール・パスワード・セッション）は Supabase Auth が管理する

profiles（ユーザープロフィール）※ Supabase Auth の users と 1:1 で紐づく
├── id (uuid) ※ auth.users.id と同値
├── display_name
├── created_at
└── updated_at

shops（店舗）
├── id (uuid)
├── user_id（登録者）
├── name
├── area（エリア）
├── address
├── phone
├── category（ジャンル）
├── price_range（価格帯: e.g. "〜1000円", "1000〜3000円"）
├── external_rating（外部サービスの評価: e.g. Google 評価）
├── business_hours（営業時間: Google の weekday_text を改行区切りのテキストで保存）
├── website_url
├── google_maps_url
├── source_url（登録元URL; URL入力時に保存）
├── photo_url
├── note（メモ）
├── created_at
└── updated_at

tags（タグマスタ）※ ユーザーが自由に作成
├── id (uuid)
├── user_id（作成者）
├── name（任意の文字列）
└── created_at

shop_tags（店舗とタグの紐付け）
├── id (uuid)
├── shop_id
└── tag_id

events（イベント）
├── id (uuid)
├── owner_user_id
├── title
├── description
├── share_token（共有リンクトークン）
├── closed_at（クローズ日時; null = オープン中）
├── created_at
└── updated_at

event_shops（イベントの候補店舗）
├── id (uuid)
├── event_id
└── shop_id

votes（投票）
├── id (uuid)
├── event_id
├── voter_name
├── user_id（会員の場合; nullable）
├── created_at

vote_choices（投票の選択内容）
├── id (uuid)
├── vote_id
└── event_shop_id（選んだ候補店舗）

```

## 7. 外部連携

### 7.1 店舗情報自動取得

| 項目 | 内容 |
|---|---|
| 利用API | Google Places API（Text Search / Place Details） |
| 店名入力時 | テキスト検索で候補一覧を表示、選択後に Place Details を取得 |
| URL入力時 | Google マップ URL から Place ID を抽出し Place Details を取得 |
| 取得項目 | 店名・住所・エリア・電話番号・カテゴリ・価格帯・評価・営業時間・Web サイト URL・写真 |
| 価格帯の変換 | Google の `price_level`（0〜4）を以下のラベルに変換して表示する: 0→「〜¥999」/ 1→「¥1,000〜¥2,999」/ 2→「¥3,000〜¥5,999」/ 3→「¥6,000〜¥9,999」/ 4→「¥10,000〜」/ 未提供→「価格帯不明」 |
| フォールバック | API エラー時は手動入力モードに切り替える |
| 費用管理 | API コール数をモニタリングし、上限アラートを設定する |
| セキュリティ | API キーはサーバー側（Cloudflare Workers の環境変数）にのみ保持し、クライアントには露出しない。ブラウザからの検索リクエストは必ず Route Handler 経由で中継する |

### 7.2 メール送信

| 項目 | 内容 |
|---|---|
| 用途 | 会員登録確認・パスワードリセット |
| 実装 | Supabase Auth の組み込みメール機能を利用（SMTP のカスタマイズも可） |

## 8. フェーズ計画

### Phase 1（v1.0 MVP）

- 認証・アカウント管理
- 店舗の記録・管理（名前/URL入力・自動補完・タグ管理・絞り込み）
- イベント作成・共有リンク
- 投票

### Phase 2（モバイル・収益化・日程調整）

- 日程調整機能（候補日設定・○△×回答・集計）
- iOS / Android ネイティブアプリ（React Native または Expo）
- フリーミアムモデル実装
  - 無料枠: マイリスト登録50件・月間イベント作成5件・カスタムタグ10種
  - 有料プラン: 無制限

### Phase 3（グローバル展開）

- 英語 UI・多言語対応（ja / en を最初にサポート）
- 地域別の外部 API 対応（Google Places 以外の選択肢）

## 9. 用語集

| 用語 | 定義 |
|---|---|
| タグ | 店舗に複数付与できる整理用ラベル。ユーザーが自由に作成・命名できる。「行きたい」「スタメン入り」「行った」は初期サジェストとして提示される |
| マイリスト | 会員が登録した店舗の全一覧 |
| イベント | 「金曜の飲み会」などの特定の集まりを単位とした、投票・日程調整のまとまり |
| 共有リンク | ゲストでもアクセス可能なイベント専用 URL |
| ゲストユーザー | 共有リンク経由でアクセスした未ログインのユーザー |
| RLS | Row Level Security。Supabase で設定するテーブル行レベルのアクセス制御 |
